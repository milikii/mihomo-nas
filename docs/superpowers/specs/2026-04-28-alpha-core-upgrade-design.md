# Alpha Core Upgrade 设计

日期：2026-04-28

## 目标

为 `minimalist` 增加一个显式、单一用途的命令，用于从官方 GitHub releases 拉取**最新 alpha 内核**，替换当前 `mihomo-core` 二进制，并在成功后自动重启 `minimalist.service`。

本设计只解决“升级到底层 alpha 内核”这一件事，不恢复旧版通道体系，不引入回滚系统，也不让部署命令隐式触发升级。

## 背景与当前真相

- 当前项目中，`minimalist` 是管理器和编排入口。
- `mihomo-core` 是底层内核，可执行路径由 `install.core_bin` 决定，默认值是 `/usr/local/bin/mihomo-core`。
- `minimalist.service` 的 `ExecStart` 直接运行 `install.core_bin` 指向的内核。
- 项目当前已明确不恢复：
  - `alpha/stable` 通道切换
  - core 回滚
  - 自动定时升级
  - 自动同步安装目录

因此，本次能力必须被严格限制在“显式升级到最新 alpha”这一最小闭环内。

## 非目标

本次不做以下内容：

- stable 通道支持
- alpha/stable 双通道切换
- 指定版本安装
- core 回滚命令
- 自动定时升级
- 多镜像源切换
- 升级 `minimalist` 自身
- 菜单入口

## 用户入口

新增显式命令：

```bash
sudo minimalist core-upgrade-alpha
```

命令行为：

1. 读取当前配置，确定 `install.core_bin`
2. 查询官方 GitHub releases API
3. 从发布列表中选择最新 alpha 发布及当前机器架构对应的 Linux 资产
4. 下载到目标目录临时文件
5. 完成解包与最小可执行校验
6. 原子替换旧 `mihomo-core`
7. 自动重启 `minimalist.service`
8. 输出升级前后版本与关键结果

该命令必须要求 `root` 运行。

## 推荐方案

采用“显式命令 + 本地原子替换 + 自动重启”的方案。

### 方案对比

#### 方案 A：显式命令 `core-upgrade-alpha`

- 优点：
  - 语义清楚
  - 不会在 `setup` / `install-self` 中隐式升级
  - 风险边界最小
  - 便于测试和文档化
- 缺点：
  - 需要用户主动执行命令

#### 方案 B：先做底层命令，再把同一能力挂到菜单

- 优点：
  - 后续可提供更友好的交互入口
- 缺点：
  - 本轮范围会扩大
  - 本质上仍然依赖方案 A 的底层实现

#### 方案 C：把升级逻辑混入 `setup` 或 `install-self`

- 优点：
  - 入口更少
- 缺点：
  - 升级动作和部署动作耦合
  - 容易在不想升级时意外升级内核
  - 不符合项目当前“显式、高可控”的方向

本设计选择方案 A。方案 B 只作为未来可能扩展，当前不实施。方案 C 明确拒绝。

## 设计细节

### 1. 发布发现

数据源固定为官方 GitHub releases API。

选择规则：

- 读取 release 列表，而不是使用仅返回 stable 的 `latest` 语义
- 只接受 `prerelease=true` 的发布
- 发布 tag 或名称必须明确带有 alpha 语义
- 只选择 Linux 资产
- 只选择与当前主机架构匹配的资产

命令应从 release 列表中选择“最新一个满足上述条件的 alpha 发布”。若列表中不存在满足条件的 alpha 发布，则直接失败，不做额外猜测。

### 2. 架构选择

架构映射仅支持当前项目边界内真实需要的 Linux 架构。

建议最小支持：

- `amd64`
- `arm64`

行为原则：

- 只做明确映射，不做模糊猜测
- 若当前架构没有可匹配资产，直接返回错误

### 3. 下载与解包

下载策略：

- 下载到 `core_bin` 同目录的临时文件
- 不直接覆盖正在使用的正式二进制
- 使用独立临时目录处理压缩包和解包产物

原因：

- 避免下载中断或解包失败破坏现网可执行文件
- 确保只有在校验通过后才进入替换阶段

### 4. 校验

最小校验要求：

- 文件存在
- 文件非空
- 文件权限可执行
- 能运行版本命令并正常输出

可接受的版本检查形式：

- `<候选二进制路径> -v`
- 或等价版本查询方式

若版本检查失败，视为新内核不可用，整个升级中止，旧文件保持不变。

### 5. 替换策略

替换采用原子 rename 流程：

1. 旧二进制先改名到同目录临时备份路径
2. 校验通过的新二进制改名到正式 `core_bin`
3. 成功后进入服务重启阶段

这里允许存在一次性 `.bak` 临时文件，但它只是替换过程中的安全垫，不属于项目能力面的“正式回滚入口”。

是否保留 `.bak`：

- 若重启成功，可删除 `.bak`
- 若重启失败，可保留 `.bak` 供人工排查，但程序不提供回滚命令，不承诺自动恢复

### 6. 服务重启

成功替换后，程序自动执行：

```bash
systemctl restart minimalist.service
```

这是本次需求的明确行为，不提供“只升级不重启”的模式。

原因：

- 新内核文件替换后，现有进程不会自动加载新版本
- 只有重启后，升级结果才真正生效

### 7. 升级后确认

重启成功后，程序应至少做以下确认：

- `systemctl is-active minimalist.service`
- 读取一次 controller 版本信息，或回退到本地版本命令输出

最终输出应包含：

- 旧 core 版本
- 新 release tag
- 命中的 asset 名称
- 替换目标路径
- 服务重启结果
- 升级后版本

## 失败处理

整体原则：只在“新文件已下载、已解包、已校验通过”后才允许触碰正式二进制；任何前置步骤失败都不得破坏旧内核。

### 前置失败

以下场景直接失败，旧内核不变：

- GitHub API 请求失败
- release 数据格式异常
- 最新 release 不是 alpha
- 找不到匹配架构资产
- 下载失败
- 解包失败
- 版本校验失败

### 替换阶段失败

若旧文件改名或新文件就位失败，命令必须立刻返回错误，并明确指出当前文件系统状态，不得输出“升级成功”。

### 重启阶段失败

这是唯一可能出现“新二进制已替换，但服务没有成功启动”的阶段。

此时命令必须：

- 返回错误
- 输出 `systemctl` 失败上下文
- 尽量附带最近的 `journalctl -u minimalist.service` 关键摘要

命令**不自动执行回滚**。原因是本项目已明确不恢复 core 回滚能力，且自动回滚会重新扩大行为边界。

## 代码落点

保持当前项目风格，不额外引入大模块。

- `internal/app`
  - 新增核心升级编排逻辑
  - 负责 root 校验、release 查询、下载、校验、替换、重启、结果输出
- `internal/cli`
  - 新增 `core-upgrade-alpha` 命令分发
  - 更新 usage

本轮不单独创建新的长期模块，除非实现时发现已有文件已明显超出可维护边界。

## 依赖策略

优先标准库实现：

- `net/http`
- `encoding/json`
- `archive/tar`
- `compress/gzip`
- `archive/zip`（如官方资产格式需要）
- `os` / `filepath`

不引入 GitHub SDK 或第三方升级框架。

原因：

- 当前需求范围小
- 减少依赖扩张
- 保持行为可读、可测

## 测试设计

### `internal/app` focused tests

至少覆盖：

- release API 返回 alpha 发布，能选中正确资产
- release 不是 alpha 时拒绝升级
- 找不到当前架构资产时失败
- 下载失败时不替换旧文件
- 解包失败时不替换旧文件
- 版本校验失败时不替换旧文件
- 替换成功后执行 `systemctl restart minimalist.service`
- 重启失败时错误上浮，并带关键上下文

### `internal/cli` tests

至少覆盖：

- `minimalist core-upgrade-alpha` 正确分发
- 非法参数时给出正确 usage

### 实机验证边界

本轮不把“联网下载官方 alpha 并实际替换本机 live core”设为必需验证门槛。

原因：

- 外网可达性与 GitHub 可用性不稳定
- 该行为直接作用 live 内核，验证风险高于普通单元测试

如果环境允许，可补一个只读 smoke：

- 仅验证可以成功查询官方最新 alpha release 元数据

## 文档更新要求

实现时需要同步更新：

- `docs/DECISIONS.md`
  - 明确“不恢复 alpha/stable 通道切换”与“允许显式升级最新 alpha core”并不冲突
- `docs/STATUS.md`
  - 记录该能力已落地及验证范围
- `docs/NEXT_STEP.md`
  - 调整下一步真相，改为“只允许显式 alpha core 升级，不引入通道/回滚”
- `README.md`
  - 补命令说明、root 要求、自动重启行为与限制

## 风险与限制

- GitHub API 或 release 命名规则变化会影响发现逻辑
- 官方发布资产命名若变化，架构匹配逻辑需要随之调整
- 自动重启意味着升级命令天然带有服务中断窗口
- 本轮不做自动回滚，因此重启失败后需要人工处理
- 本轮不保证所有 Linux 架构，超出映射范围的机器应明确失败

## 成功判定

完成实现后，应满足以下条件：

1. `minimalist core-upgrade-alpha` 可由 CLI 正确分发
2. 能识别官方最新 alpha release 并选中匹配资产
3. 在下载、解包、校验任一步骤失败时，不破坏旧 `mihomo-core`
4. 成功替换后会自动重启 `minimalist.service`
5. 文档同步更新为当前真相
6. 相关 focused tests 通过

## 结论

本设计引入的是一个**显式、单用途、可控**的 alpha 内核升级入口，而不是恢复旧版核心通道系统。

能力边界保持为：

- 可以显式升级到最新 alpha
- 自动重启服务使升级立即生效
- 不提供通道切换
- 不提供回滚
- 不引入自动升级

这与当前项目“质量硬化 + 最小实现 + 边界收口”的方向一致。
